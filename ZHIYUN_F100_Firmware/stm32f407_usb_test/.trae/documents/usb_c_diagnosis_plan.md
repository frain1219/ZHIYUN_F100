# STM32F407 USB-C 口无法识别诊断计划

## 问题概述

* F407 开发板上的 TYPE-C 已连接到电脑，但电脑无法识别设备

* WCH（DAP-Link）已连接并正常工作（/dev/cu.usbmodemBC66E5A6A20C1）

* 当前固件已通过 WCH 烧录到 F407

## 当前固件配置分析

### USB 配置（已确认）

* **引脚**：PA11 (USB\_DM) / PA12 (USB\_DP)

* **PHY**：嵌入式 PHY (PCD\_PHY\_EMBEDDED)

* **VBUS 感知**：已禁用 (vbus\_sensing\_enable = DISABLE)

* **时钟**：HSE 25MHz → PLLQ=7 → USB 48MHz

### 代码结构

* 使用 STM32 HAL USB Device 库

* CDC 类实现

* LED 指示：500ms 慢闪 = 未枚举，200ms 快闪 = 已连接

***

## 可能的原因分析

### 1. 硬件连接问题（高概率）

#### 1.1 USB-C 口引脚映射问题（我已确认PA11 PA12连接的就是USB）

**问题**：开发板的 USB-C 口可能不是直接连接到 PA11/PA12

**验证方法**：

* 检查开发板原理图，确认 USB-C 口的 D+/D- 连接

* 使用万用表测量 USB-C 口 D+/D- 与 PA11/PA12 的连通性

* 检查 USB-C 口的 CC1/CC2 引脚配置（需要 5.1k 下拉电阻）

#### 1.2 USB-C CC 引脚配置

**问题**：USB-C 需要 CC 引脚正确配置才能被主机识别

**验证方法**：

* 检查 USB-C 母座 CC1/CC2 是否有 5.1kΩ 下拉电阻到 GND

* 如果没有，USB-C 口将无法被电脑识别为 USB 设备

### 2. 时钟问题（中概率）

#### 2.1 HSE 晶振问题

**问题**：如果外部 25MHz 晶振未正常工作，USB 时钟将不正确

**验证方法**：

* 观察 LED 闪烁频率

  * 50ms 快闪 = HSE 失败（时钟初始化错误）

  * 500ms 慢闪 = HSE 正常但 USB 未枚举

* 使用示波器测量 OSC\_IN/OSC\_OUT 引脚

#### 2.2 USB 48MHz 时钟精度

**问题**：USB 需要 ±0.25% 的时钟精度

**验证方法**：

* 确认 PLLQ=7 的计算：25MHz / 25 \* 336 / 7 = 48MHz ✓

* 检查 HSE\_VALUE 是否与实际晶振匹配

### 3. USB 外设配置问题（中概率）

#### 3.1 GPIO 配置冲突

**问题**：PA11/PA12 可能被其他功能占用

**验证方法**：

* 检查是否有其他代码配置了 PA11/PA12

* 确认 USB 初始化前 PA11/PA12 未被其他外设使用

#### 3.2 中断优先级问题

**问题**：USB 中断可能被其他中断阻塞

**验证方法**：

* 检查 OTG\_FS\_IRQn 中断优先级配置（当前为 5, 0）

### 4. 开发板型号差异（高概率）

#### 4.1 USB 接口选择

**问题**：部分 F407 开发板使用 USB\_OTG\_HS 而非 USB\_OTG\_FS

**验证方法**：

* 检查开发板型号和原理图

* 确认 USB-C 口连接的是 OTG\_FS (PA11/PA12) 还是 OTG\_HS

#### 4.2 参考例程对比

**问题**：参考例程使用旧版 USB 库，配置方式不同

**验证方法**：

* 对比参考例程 `例22 USB读卡器(Slave)` 的 USB 配置

* 参考例程仅配置 PA11/PA12，未配置 PA9/PA10

## 诊断步骤

### 步骤 1：LED 状态观察

1. 上电后观察 LED 闪烁频率
2. 判断时钟初始化是否成功

   * 50ms 快闪 → HSE 失败

   * 500ms 慢闪 → HSE 正常，USB 未枚举

### 步骤 2：硬件连通性检查

1. 使用万用表测量 USB-C 口 D+ 与 PA12 的连通性
2. 使用万用表测量 USB-C 口 D- 与 PA11 的连通性
3. 检查 USB-C 口 CC1/CC2 是否有下拉电阻

### 步骤 3：时钟验证

1. 如果 HSE 失败，检查外部晶振
2. 如果 HSE 正常，使用示波器测量 USB 时钟

### 步骤 4：参考例程测试

1. 烧录参考例程 `例22 USB读卡器(Slave)` 的 hex 文件
2. 观察电脑是否能识别设备
3. 如果能识别，对比配置差异

### 步骤 5：USB 描述符验证

1. 使用 USB 分析工具（如 Wireshark + USBPcap）捕获 USB 枚举过程
2. 检查是否有任何 USB 通信

## 可能的解决方案

### 方案 A：硬件修改

如果 USB-C 口的 D+/D- 未连接到 PA11/PA12：

1. 飞线连接 USB-C D+ → PA12，D- → PA11
2. 添加 CC1/CC2 下拉电阻（5.1kΩ）

### 方案 B：修改固件配置

如果 USB-C 口连接到其他引脚：

1. 修改 usbd\_conf.c 中的 GPIO 配置
2. 如果使用 OTG\_HS，修改 USB 外设初始化

### 方案 C：使用参考例程的 USB 库

如果 HAL 库配置有问题：

1. 移植参考例程的旧版 USB 库
2. 对比配置差异

## 🔴 关键发现：HSE 晶振频率不匹配！

### 参考例程配置
- **HSE_VALUE = 8MHz**（`stm32f4xx.h` 第123行）
- **PLL_M = 8**
- 时钟计算：8MHz / 8 * 336 / 2 = 168MHz（SYSCLK），8MHz / 8 * 336 / 7 = 48MHz（USB）✓

### 当前固件配置
- **HSE_VALUE = 25MHz**（`platformio.ini` 中 `-DHSE_VALUE=25000000`）
- **PLL_M = 25**
- 时钟计算：25MHz / 25 * 336 / 2 = 168MHz（SYSCLK），25MHz / 25 * 336 / 7 = 48MHz（USB）✓

### 问题分析
**如果开发板实际使用的是 8MHz 晶振，而固件配置为 25MHz：**
- VCO输入 = 8MHz / 25 = 0.32MHz（远低于 PLL 最小输入频率 1MHz）
- **PLL 将无法正常工作**
- **HSE 可能无法启动**，导致系统时钟异常
- USB 48MHz 时钟将不正确

### 验证方法
1. **观察 LED 闪烁频率**：
   - 50ms 快闪 = HSE 失败（时钟初始化错误）
   - 500ms 慢闪 = HSE 正常但 USB 未枚举

2. **检查开发板晶振**：
   - 查看开发板上的晶振型号
   - 用示波器测量 OSC_IN/OSC_OUT 引脚频率
